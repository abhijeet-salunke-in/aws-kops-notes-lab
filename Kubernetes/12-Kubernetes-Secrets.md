# Kubernetes Secrets 

## Introduction

Applications require data to run.

Examples:

* Database username
* Database password
* API tokens
* JWT secrets
* AWS Access Keys
* SSH Keys
* TLS Certificates
* Docker Registry Credentials

Example:

```text
Database:
Username = admin
Password = admin123

AWS:
AccessKey = AKIAxxxx
SecretKey = xxxxxxxxx

Payment Gateway:
API_KEY = abc123xyz
```

These values are sensitive.

If someone gets them:

* Database can be hacked
* Cloud resources can be compromised
* Application can be abused
* Data can be stolen

Because of this Kubernetes provides a special object called Secret.

---

# What is a Secret?

A Secret is a Kubernetes object used to store and manage sensitive information separately from application code and Pod definitions.

Secret stores:

```text
Passwords
Tokens
Certificates
SSH Keys
API Keys
Cloud Credentials
```

Instead of putting sensitive values directly inside Pod YAML, Kubernetes stores them in Secrets and injects them when needed.

---

# Why Do We Need Secrets?

Consider this Pod:

```yaml
env:
- name: DB_USER
  value: admin

- name: DB_PASSWORD
  value: admin123
```

Problems:

### Security Risk

Anyone with YAML access can see passwords.

### GitHub Exposure

Developers may accidentally push credentials.

### Difficult Management

Changing password requires editing all manifests.

### Compliance Issues

Violates security standards.

---

## Better Approach

Store credentials separately:

```text
Secret
    |
    |
    +---- username
    |
    +---- password
```

Application consumes Secret.

Benefits:

* Centralized management
* Better security
* Easier updates
* Industry standard

---

# Real World Example

Suppose an application connects to MySQL.

Application needs:

```text
DB_HOST=mysql-service
DB_USER=root
DB_PASSWORD=root123
```

Configuration:

```text
DB_HOST -> ConfigMap
DB_USER -> Secret
DB_PASSWORD -> Secret
```

Why?

Because host is not sensitive.

Password is sensitive.

---

# ConfigMap vs Secret

This is one of the most important interview questions.

| Feature   | ConfigMap           | Secret               |
| --------- | ------------------- | -------------------- |
| Purpose   | Store configuration | Store sensitive data |
| Data Type | Non-sensitive       | Sensitive            |
| Storage   | Plain text          | Base64 encoded       |
| Security  | Low                 | Higher               |
| Examples  | App Name, URL, Port | Password, Token      |
| Usage     | Configuration       | Credentials          |

Example:

ConfigMap:

```yaml
data:
  APP_COLOR: blue
  PORT: "8080"
```

Secret:

```yaml
data:
  PASSWORD: YWRtaW4xMjM=
```

---

# Important Note About Secret Security

Many beginners think Secret is encrypted.

It is NOT encrypted by default.

Kubernetes stores Secret as Base64 encoded.

Example:

```text
admin123
```

Encoded:

```text
YWRtaW4xMjM=
```

Anyone can decode:

```bash
echo YWRtaW4xMjM= | base64 -d
```

Output:

```text
admin123
```

Therefore:

```text
Secret ≠ Encryption
Secret = Better Management + Access Control
```

# Ways to Create Secrets

Your trainer mentioned 4 methods.

These are correct.

## Method 1

Create Secret using Literal Values

## Method 2

Create Secret using File

## Method 3

Create Secret using Environment File

## Method 4

Create Secret using Folder/Directory

---

# METHOD 1: Secret Using Literal

---

## When Is It Used?

Quick testing

Labs

Interview Practice

Not recommended for production.

---

## Step 1 Create Secret

```bash
kubectl create secret generic db-secret \
--from-literal=username=admin \
--from-literal=password=admin123
```

Explanation:

```text
generic = Opaque Secret
username = key
admin = value
```

---

## Step 2 Verify

```bash
kubectl get secret
```

Output:

```text
NAME        TYPE
db-secret   Opaque
```

---

## Step 3 Describe

```bash
kubectl describe secret db-secret
```

Output:

```text
username
password
```

Values hidden.

---

## Step 4 View YAML

```bash
kubectl get secret db-secret -o yaml
```

Output:

```yaml
data:
  username: YWRtaW4=
  password: YWRtaW4xMjM=
```

---

## Decode

```bash
echo YWRtaW4= | base64 -d
```

Output:

```text
admin
```

---

```

```

# METHOD 4: Secret Using Folder

---

## Why Use It?

Suppose directory contains:

```text

```

```text

```

# Using Secret Inside Pods

There are two major ways.

---

# Method A: Environment Variables

Most commonly used.

Pod:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: secret-pod
spec:
  containers:
  - name: app
    image: nginx

    env:
    - name: USERNAME
      valueFrom:
        secretKeyRef:
          name: db-secret
          key: username

    - name: PASSWORD
      valueFrom:
        secretKeyRef:
          name: db-secret
          key: password
```

Apply:

```bash
kubectl apply -f pod.yaml
```

---

## Verify

```bash
kubectl exec -it secret-pod -- sh
```

Inside:

```bash
printenv
```

Output:

```text
USERNAME=admin
PASSWORD=admin123
```

---

# Method B: Mount Secret as Volume

Preferred for certificates and keys.

Pod:

```yaml
volumes:
- name: secret-vol
  secret:
    secretName: db-secret
```

Mount:

```yaml
volumeMounts:
- name: secret-vol
  mountPath: /etc/secret
```

---

Files Created:

```text
/etc/secret/username
/etc/secret/password
```

Read:

```bash
cat /etc/secret/password
```

# Advantages of Secrets

### Security Separation

Credentials separated from application code.

### Centralized Management

Update once.

Use everywhere.

### Better Access Control

RBAC can restrict access.

### Production Standard

Used by all enterprise Kubernetes environments.

---

# Limitations of Secrets

### Not Encrypted

Only Base64 encoded.

### Visible to Cluster Admins

Admins can decode values.

### Namespace Scoped

Cannot directly share across namespaces.

### Pod Restart Sometimes Needed

Applications may need restart after Secret update.

---
